# TrustScore — Financial Fraud Risk Detector (Real Data: PaySim + SMOTE)

An end-to-end fraud detection project built on the real, widely-used
**PaySim** dataset — 6.36 million synthetic mobile-money transactions
modeled on real transaction logs from a mobile money provider operating
in 14 countries (originally published on Kaggle as "PaySim1").

---

## 1. Problem Statement

Financial institutions process millions of transactions daily.
Fraudulent transfers/cash-outs are extremely rare (in this dataset,
just **0.13%** of all transactions — 8,213 out of 6,362,620) but
costly, and manual review of every transaction is impossible. The goal
is to build a model that scores each transaction's fraud probability
so only the highest-risk transactions get flagged for review.

## 2. Proposed Solution

1. **Data**: PaySim — real, industry-standard fraud detection benchmark
   dataset, downloaded from Hugging Face
   (`purulalwani/Synthetic-Financial-Datasets-For-Fraud-Detection`).
2. **SQL layer**: Load into SQLite, use SQL to discover that fraud
   *only* occurs in `TRANSFER` and `CASH_OUT` transaction types — a
   key insight that shrinks the modeling problem from 6.36M rows down
   to ~135K relevant rows.
3. **Feature engineering**: Derive balance-mismatch features
   (`errorBalanceOrig`, `errorBalanceDest`) — the gap between what an
   account balance *should* be after a transaction and what it
   actually is. This turned out to be, by far, the strongest fraud
   signal (44.4% feature importance).
4. **Class imbalance handling**: Applied **SMOTE** (Synthetic Minority
   Oversampling) to the training set only — real fraud rows are used
   to generate new synthetic fraud examples until the training classes
   are balanced 1:1. The test set is left untouched at its real,
   imbalanced ratio so evaluation metrics stay honest.
5. **ML Model**: Logistic Regression (baseline) + Random Forest (main
   model), evaluated with precision/recall/ROC-AUC/accuracy.
6. **Dashboard**: Export scored transactions to CSV for Power BI.

## 4. Setup & Run (using `uv`)

**Step 1 — Download the data manually** (one-time, ~274MB):
`https://huggingface.co/datasets/purulalwani/Synthetic-Financial-Datasets-For-Fraud-Detection/resolve/main/financial_txns.parquet?download=true`
Place the file at `data/financial_txns.parquet`.

**Step 2 — Set up and run:**

```bash
cd fraud_project
uv venv
.venv\Scripts\activate          # Windows
# source .venv/bin/activate     # macOS/Linux

uv pip install -r requirements.txt
uv run python data/download_data.py
uv run python analysis_and_model.py
```

Always run via the terminal with `uv run` — not the VS Code Run
button, which can execute from the wrong working directory.

## 5. Actual Results (real run, full pipeline with SMOTE)

**Fraud rate by transaction type (from SQL):**

| Type | Fraud Rate |
|---|---|
| TRANSFER | 14.38% |
| CASH_OUT | 3.86% |
| PAYMENT / DEBIT / CASH_IN | 0% |

**Class balance before/after SMOTE (training set only):**

| | Normal | Fraud |
|---|---|---|
| Before SMOTE | 95,245 | 6,160 |
| After SMOTE | 95,245 | 95,245 |

**Model comparison (test set, 33,802 rows — untouched, real ratio):**

| Model | Accuracy | Fraud Precision | Fraud Recall | ROC-AUC |
|---|---|---|---|---|
| Logistic Regression | 0.9389 | 0.50 | 0.88 | 0.9760 |
| **Random Forest** | **0.9998** | **1.00** | **1.00** | **0.9997** |

Random Forest's confusion matrix: only **1 false positive** and
**5 false negatives** out of 33,802 test transactions.

**Top fraud predictor (Random Forest feature importance):**
1. `errorBalanceOrig` (44.4%) — sender account balance mismatch
2. `oldbalanceOrg` (18.9%) — sender's starting balance
3. `newbalanceOrig` (10.5%) — sender's ending balance
4. `newbalanceDest` (9.1%)
5. `amount` (5.7%)

## 7. Limitations & Next Steps

- **Near-perfect scores are a known property of this specific
  dataset**, not a general claim about fraud detection. PaySim
  generates fraud through a deterministic accounting-mismatch rule
  (`errorBalanceOrig`), so a tree-based model can learn it almost
  exactly. Real-world fraud is much messier, so production accuracy
  would be meaningfully lower.
- SMOTE was applied only to the training set, correctly avoiding data
  leakage into the test set — worth calling out explicitly, since it's
  a common mistake.
- No network/graph features (e.g., how many distinct accounts a given
  `nameDest` receives from) — production systems typically add
  graph-based features to catch mule-account rings.
- Real-time scoring would need the model served via an API, not a
  batch CSV export.
